# Phase 3 — Terminal UI Design Contract

> Visual and interaction contract for CLI terminal output. Generated by gsd-ui-researcher. This phase uses Ink (React for terminals), chalk, and Box/Text components — not a web frontend.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (CLI — no shadcn) |
| Preset | not applicable |
| Component library | Ink 6 (Box, Text) |
| Icon library | none (Unicode symbols only) |
| Font | Terminal default (monospace, user-controlled) |

---

## Terminal Color Palette

Terminal colors use ANSI names via chalk/Ink. No hex values — the user's terminal theme controls actual rendered color.

| Role | Ink/Chalk Color | Usage |
|------|-----------------|-------|
| Success | `green` | Success confirmation messages, active context indicators |
| Error | `red` | Error messages, error borders |
| Accent / Command hints | `cyan` | Command suggestions in help text, empty states |
| Dim / Secondary | `dimColor` (Ink prop) | Parenthetical info, helper text, "(not set)" labels |
| Bold | `bold` (Ink prop) | Table headers, entity names in success messages |
| Default | terminal default | Table data cells, general body text |

Reserved color usage:
- `green` — success confirmations only (create, rename, delete success, active context)
- `red` — error display only (API errors, validation errors, bordered error boxes)
- `cyan` — command hints only (suggested next commands in empty states and post-action hints)
- `dimColor` — secondary/helper text only (suggestions, labels, not-set indicators)

---

## Spacing (Terminal Units)

Terminal spacing uses character cells, not pixels. Ink `padding` and `margin` props use integer character counts.

| Token | Value (chars) | Usage |
|-------|---------------|-------|
| inline | `paddingX={1}` | Inside bordered boxes, between border and text |
| block | `paddingY={1}` | Inside bordered boxes, vertical breathing room |
| section | `marginTop={1}` | Between output sections (e.g., success box and hint) |
| outer | `padding={1}` | Outermost component wrapper |

Established by existing components (ErrorDisplay, AuthSuccess, AuthStatus). All new components follow these same values.

---

## Typography (Terminal)

Terminal text has no font-size control. "Typography" is expressed through Ink `Text` props.

| Role | Ink Props | Usage |
|------|-----------|-------|
| Body | `<Text>` (no props) | Table data cells, general output |
| Bold | `<Text bold>` | Table column headers, entity names in confirmations |
| Dim | `<Text dimColor>` | Helper text, suggestions, "(not set)" |
| Success | `<Text color="green" bold>` | Success message inside bordered box |
| Error | `<Text color="red" bold>` | Error message inside bordered box |
| Command hint | `<Text color="cyan">` | Suggested commands (e.g., `kanbambam workspace create`) |

## Component Inventory

### 1. Table (NEW)

Reusable bordered table for all list commands (workspace list, board list, column list).

**Ink structure:**
```
<Box flexDirection="column" borderStyle="round" paddingX={1}>
  <Box>  <!-- header row -->
    <Box width={N}><Text bold>{column label}</Text></Box>
    ...
  </Box>
  <Box>  <!-- separator row -->
    <Text dimColor>{repeated dash chars}</Text>
  </Box>
  <Box>  <!-- data row, repeated -->
    <Box width={N}><Text>{cell value}</Text></Box>
    ...
  </Box>
</Box>
```

**Column widths by entity:**

| Entity | Columns | Widths |
|--------|---------|--------|
| Workspace | ID, Name | 20, 30 |
| Board | ID, Name | 20, 30 |
| Column | ID, Name, Position | 20, 24, 10 |

**Border style:** `round` (matches existing ErrorDisplay, AuthSuccess, AuthStatus pattern).

**Plain mode fallback:** Aligned columns using string padding. No borders, no color. Example:
```
ID                   Name
wks_abc123           My Workspace
wks_def456           Another Workspace
```

### 2. SuccessMessage (NEW)

One-line success confirmation after create, rename, delete operations.

**Ink structure:**
```
<Box padding={1}>
  <Text color="green" bold>{message}</Text>
</Box>
```

No bordered box for success messages — keep them lightweight. The bordered box pattern from AuthSuccess is for multi-line status displays. Single-line confirmations use plain green bold text.

**Plain mode fallback:** Print message string without color.

### 3. EmptyState (NEW)

Displayed when a list command returns zero results.

**Ink structure:**
```
<Box padding={1}>
  <Text dimColor>{message} Run </Text>
  <Text color="cyan">{suggested command}</Text>
  <Text dimColor> to create one.</Text>
</Box>
```

**Plain mode fallback:** Print message string without color.

### 4. ContextDisplay (NEW)

Shows active workspace and board context.

**Ink structure:**
```
<Box flexDirection="column" padding={1}>
  <Box>
    <Text bold>Workspace: </Text>
    <Text>{name}</Text>
    <Text dimColor> ({id})</Text>
  </Box>
  <Box>
    <Text bold>Board:     </Text>
    <Text>{name}</Text>
    <Text dimColor> ({id})</Text>
  </Box>
</Box>
```

When a value is not set:
```
<Text dimColor>(not set)</Text>
```

**Plain mode fallback:**
```
Workspace: My Project (wks_abc123)
Board:     Sprint Board (brd_def456)
```

### 5. ConfirmPrompt (NOT an Ink component)

Uses Node.js `readline/promises`. NOT Ink — avoids render loop conflicts.

**Format:** `{action description}? [y/N] `

Examples:
- `Delete workspace "My Project" (wks_abc123)? [y/N] `
- `Delete board "Sprint Board" (brd_def456)? [y/N] `
- `Delete column "Done" and all its items? [y/N] `

Default is No (uppercase N). Only `y` or `Y` confirms.

### 6. ErrorDisplay (EXISTING)

Already implemented in `apps/cli/src/components/ErrorDisplay.tsx`. Red bordered box with optional suggestion.

**No changes needed.** New commands reuse this component for API errors and validation errors.

### 7. Spinner (EXISTING)

Already implemented in `apps/cli/src/components/Spinner.tsx`.

**No changes needed.** Use for multi-step operations (board create with default columns, column delete with item migration).

## Copywriting Contract

### Success Messages

| Command | Success Copy |
|---------|-------------|
| workspace create | `Created workspace {name} ({id})` |
| workspace rename | `Renamed workspace to {name} ({id})` |
| workspace delete | `Deleted workspace {name} ({id})` |
| board create | `Created board {name} ({id}) with columns: To Do, In Progress, Done` |
| board rename | `Renamed board to {name} ({id})` |
| board delete | `Deleted board {name} ({id})` |
| column add | `Added column {name} ({id})` |
| column rename | `Renamed column to {name} ({id})` |
| column reorder | `Moved column {name} to position {N}` |
| column delete | `Deleted column {name} ({id})` |
| context set | `Context set to workspace {name} ({id})` or `Context set to board {name} ({id}) in workspace {wsName}` |
| context clear | `Context cleared` |

### Empty State Messages

| Command | Empty Copy |
|---------|-----------|
| workspace list | `No workspaces yet. Run kanbambam workspace create <name> to create one.` |
| board list | `No boards in this workspace. Run kanbambam board create <name> to create one.` |
| column list | `No columns on this board. Run kanbambam column add <name> to add one.` |

### Error Messages

| Scenario | Error Copy |
|----------|-----------|
| No workspace context | `No workspace specified. Use --workspace <id> or set context with: kanbambam context set --workspace <id>` |
| No board context | `No board specified. Use --board <id> or set context with: kanbambam context set --board <id>` |
| Not authenticated | `Not logged in. Run kanbambam auth login to authenticate.` |
| Entity not found (404) | `{Entity} not found. It may have been deleted. Run kanbambam context clear if your context is stale.` |
| Column has items (no flag) | `Column "{name}" has {N} items. Use --move-to <column-id> to relocate them, or --force to delete them.` |
| API error (generic) | `{API error message}` (pass through from API response error.message) |
| Network error | `Could not reach the API. Check your connection and try again.` |

### Destructive Confirmation Prompts

| Command | Confirmation Copy |
|---------|------------------|
| workspace delete | `Delete workspace "{name}" ({id})? This will delete all boards, columns, and items. [y/N]` |
| board delete | `Delete board "{name}" ({id})? This will delete all columns and items. [y/N]` |
| column delete (empty) | No prompt (deletes immediately, or with `--force`) |
| column delete (has items, --force) | `Delete column "{name}" and its {N} items? [y/N]` |

## Output Mode Contract

Every command must support three output modes per the existing `detectOutputMode()` pattern:

| Mode | Trigger | Behavior |
|------|---------|----------|
| `ink` | TTY stdout, no `--json` flag | Render Ink component with colors, borders, formatting |
| `json` | `--json` flag passed | Output `{ data, meta }` envelope via `outputJson()` |
| `plain` | Non-TTY stdout (piped), no `--json` | Plain text, no color, no borders, aligned columns |

**JSON output envelope:**
- Create/rename: `{ data: { id, name, ... }, meta: { requestId } }`
- List: `{ data: [...], meta: { total, cursor } }`
- Delete: `{ data: { deleted: true, id }, meta: {} }`
- Context show: `{ data: { workspace: { id, name } | null, board: { id, name } | null }, meta: {} }`
- Context set/clear: `{ data: { workspace: { id, name } | null, board: { id, name } | null }, meta: {} }`

## Interaction Patterns

### Delete Flow
1. Check `--force` flag. If set, skip to step 4.
2. For column delete: check if column has items. If yes and no `--move-to`/`--force`, show error and exit.
3. Show confirmation prompt via readline (not Ink). Wait for y/N.
4. If confirmed (or `--force`): execute API call(s).
5. Render success message via Ink/plain/JSON.

### Context Resolution Flow
1. Check explicit flag (`--workspace`, `--board`).
2. If no flag, read from `conf` stored context.
3. If neither exists, show error with usage hint and exit (non-zero exit code).

### Board Create Flow (multi-step)
1. Create board via API.
2. Create "To Do" column.
3. Create "In Progress" column.
4. Create "Done" column.
5. Show single success message: `Created board {name} ({id}) with columns: To Do, In Progress, Done`.
6. If any column creation fails after board creation, show partial success: `Created board {name} ({id}). Warning: failed to create default column "{colName}".`

### Column Delete with --move-to Flow
1. List items in the column via API.
2. Move each item to target column (parallel, max 5 concurrent).
3. Delete the column via API.
4. Show success: `Moved {N} items to column {targetName} and deleted column {name}`.
